Introdução

✦ Introdução

O que é o Kiro e como funciona o evento

Este é um evento da AWS, e o Kiro é a ferramenta principal — um editor de código com inteligência artificial que a AWS criou. Em vez de escrever tudo em C# na mão, vocês vão conversar com ele em inglês descrevendo o que querem, e ele lê o projeto e escreve ou edita os arquivos, explicando o que mudou.

Como funciona o dia

Vocês vão trabalhar em equipe, cada pessoa com um papel (seção anterior), construindo um jogo 2D no Unity que conversa com um hardware físico (ESP32 + OLED). O objetivo mínimo e as personalizações estão nos checklists deste guia — sigam a ordem, testando cada etapa antes de seguir pra próxima.

Times e pessoa de suporte

O evento é dividido em times, e cada time tem uma pessoa de suporte dedicada durante o dia. Sempre que travarem em algo — um erro que não passa, uma dúvida sobre o passo a passo, o repositório que não abre — chamem o suporte do seu time. Ninguém precisa resolver tudo sozinha.

Todo mundo está aprendendo junto

Não tem problema não saber algo de antemão — ninguém precisa chegar sabendo Unity, C# ou ESP32. O guia foi feito pra isso: seguir o passo a passo, testar, e pedir ajuda quando travar.

Onde encontrar este guia

O link e o QR code deste material são compartilhados pela organização no início do evento. O documento oficial também está linkado no rodapé desta página.

🎮 Introdução

Que jogo é esse, afinal

Vocês vão construir um jogo 2D de plataforma — daqueles em que a jogadora controla um personagem que anda de um lado pro outro e pula. A grande sacada é que esse jogo conversa com um hardware físico: enquanto a jogadora joga na tela, uma telinha (o OLED) ali do lado mostra as reações do personagem em tempo real.

O objetivo do jogo

A jogadora atravessa a fase coletando todos os itens e desviando de obstáculos. Quando o último item é coletado (ou ela chega no ponto final), aparece o painel de fim de jogo. É um jogo de "pegue tudo pra vencer".

As regras principais

O que aconteceEfeito no jogoReação no OLED
Pega um item coletávelO item some e é contadoCarinha feliz (T)
Encosta num obstáculoDispara a reação de danoCarinha assustada (D)
Pega o último item / chega no finalPainel de fim, jogo pausaCarinha feliz (T)
Jogo rodando normal—Carinha neutra (I)
A lógica que liga tudo

Guarde este fluxo, porque é o coração do projeto: a jogadora faz algo no jogo → o Unity percebe → manda uma letra pela USB → o ESP32 recebe → a carinha muda no OLED. Todas as fases técnicas mais adiante existem pra construir alguma parte desse caminho.

O mínimo que o jogo precisa ter

1 fase jogável, 1 personagem controlável (pelo controle do ESP32), 3 itens coletáveis, 3 obstáculos, e a comunicação com o OLED reagindo (carinhas neutra, feliz e de dano). A lista completa pra conferir na entrega está na seção 33.

Antes de tudo

01 Preparação

O que precisa estar instalado

Antes do evento, é bom já deixar tudo instalado na máquina pra não perder tempo no dia.

Em palavras simples

O Git serve pra baixar o projeto do GitHub. Baixar uma cópia do projeto pra sua máquina tem um nome: clonar o repositório. Pelo GitHub Desktop, o caminho é entrar no repo, clicar em Code e clonar.

02 Preparação

Clonar o repositório do GitHub

O repositório do projeto é:

https://github.com/Connect-Byte-Hangout/unity-template-aws-game

Sugestão de pasta pra salvar: C:\Projetos\ConnectByte\

Atenção

Evite salvar dentro do OneDrive ou em pastas com nomes muito longos — o Unity fica chato com caminho sincronizado.

Depois de clonar, deve aparecer uma pasta chamada unity-template-aws-game — e essa pasta já é o projeto Unity, contendo:

Assets/ Packages/ ProjectSettings/ .gitignore

Duas formas de clonar. Escolha a que for mais confortável pra você.

Opção 1 — Pelo Git Bash (por comando)

Passo 1. No Windows, crie a pasta C:\Projetos.
Passo 2. Clique com o botão direito dentro dela e escolha Open Git Bash here.
Passo 3. Rode os comandos abaixo (um por linha, apertando Enter em cada um):

git bash
git clone https://github.com/Connect-Byte-Hangout/unity-template-aws-game.git
cd unity-template-aws-game
ls

O git clone baixa o projeto, o cd entra na pasta baixada, e o ls lista o que tem dentro. Se deu tudo certo, o ls deve mostrar Assets, Packages e ProjectSettings — os três sinais de que essa pasta é o projeto Unity.

Se der erro

Se aparecer Repository not found ou 404 Not Found, geralmente é uma destas duas coisas: o link está digitado errado, ou o repositório ainda está privado (só quem foi convidado consegue acessar). Antes de ficar tentando sozinha, peça para a pessoa de suporte do seu time confirmar: (1) se o repositório já está marcado como público no GitHub, e (2) se o link é exatamente este. Assim você não perde tempo debugando um problema que não é da sua máquina.

Opção 2 — Pelo Kiro

Abra o Kiro dentro da pasta onde você quer salvar o projeto (ex: C:\Projetos) e peça:

kiro · clonar repositório
Clone the repository at https://github.com/Connect-Byte-Hangout/unity-template-aws-game.git
into the current folder.
After cloning, list the contents of the resulting folder so I can confirm it worked.

O Kiro roda o git clone por você e mostra o resultado.

Se der erro

Mesmo problema da opção do Git Bash: o erro 404/Repository not found costuma ser link errado ou repositório ainda privado. Peça para a pessoa de suporte do seu time confirmar se ele já está público antes de insistir tentando de novo.

04 Preparação

Abrir o projeto no Unity Hub

O Unity Hub serve pra gerenciar e abrir projetos, além de instalar versões do Editor.

  1. Abra o Unity Hub.
  2. Clique em Add project from disk.
  3. Escolha a pasta raiz que você clonou (ex: C:\Projetos\unity-template-aws-game) — não precisa entrar em nenhuma subpasta.
  4. Confira se essa pasta tem Assets, Packages, ProjectSettings.
  5. Clique pra abrir.
É normal

Na primeira vez o Unity pode demorar e mostrar Importing Assets, Compiling Scripts, Resolving Packages. Respira — isso faz parte.

05 Preparação

Se o Unity pedir outra versão

Este projeto foi feito no Unity 6.5 (versão 6000.5.1f1). O ideal é instalar exatamente essa versão pelo Unity Hub antes de abrir — assim ele não reclama.

Como instalar a versão certa

No Unity Hub, aba Installs → Install Editor → procure 6000.5.1f1. Se não aparecer na lista, clique em "skip"/"outras versões" ou baixe pelo arquivo de versões da Unity.

Se a versão que aparecer na sua máquina for quase igual — por exemplo 6000.5.1f2 em vez de 6000.5.1f1, ou 6000.5.2f1 — pode manter e abrir mesmo assim. A diferença no finalzinho (o f1/f2) é só uma pequena correção de patch e não atrapalha o projeto. O Unity pode mostrar um aviso perguntando se quer abrir com essa versão; é só aceitar.

Regra segura

Nunca abra com uma versão mais antiga que a do projeto. Se foi feito no Unity 6.5, não tente abrir no Unity 2022 — pode corromper ou dar erro que assusta à toa.

Conhecendo o Unity

06 Unity

As janelas do Unity

As próximas três seções (6 a 8) são só pra conhecer e testar se o projeto abre e funciona — vocês ainda não vão criar nada agora. O "mão na massa" começa na Fase 1, um pouco mais adiante.

O Unity abre em várias janelas ao mesmo tempo — aqui está pra que serve cada uma que você vai usar sem parar.

JanelaPra que serve
HierarchyLista de tudo que está na cena (ex: Player, Ground, Canvas, GameManager, ESP32SerialReader).
SceneOnde você monta a fase visualmente (arrasta chão, personagem, obstáculos e itens).
GameOnde você vê o jogo rodando quando aperta Play.
ProjectOnde ficam os arquivos (Assets, Scripts, Sprites, Prefabs, Scenes).
InspectorConfigurações do objeto selecionado (Transform, Sprite Renderer, Rigidbody2D, Collider2D, scripts).
ConsoleErros e avisos. Erro vermelho na Console é algo pra olhar.
Dois conceitos que aparecem toda hora

GameObject é qualquer "coisa" dentro da cena — o personagem, o chão, uma luz, uma caixa vazia. Tudo que aparece na Hierarchy é um GameObject. Component é uma "peça" que você anexa num GameObject pra dar a ele um comportamento — por exemplo, Sprite Renderer (mostra uma imagem), Rigidbody2D (física) e Collider2D (colisão) são components. Um GameObject pode ter vários components ao mesmo tempo — é isso que aparece listado no Inspector quando você clica nele.

07 Unity

Vamos fazer um teste: abrir a cena

Antes de mexer em qualquer coisa, vamos confirmar que o projeto abre e funciona direitinho. Isso evita descobrir um problema de configuração só depois de já ter feito mudanças.

No painel Project, procure uma pasta chamada Scenes (pode estar em Assets/Scenes ou Assets/_Project/Scenes).

Dentro dela, procure arquivos .unity. Nomes possíveis: MainScene, GameScene, Level01, SampleScene.

Na dúvida

Dê dois cliques na cena principal. Se não souber qual é, tente a que tiver nome mais parecido com MainScene ou SampleScene.

08 Unity

Rodar a cena

Com a cena aberta, é hora do primeiro teste real: apertar Play e ver se o jogo aparece.

  1. Clique no botão de Play (▶) no topo do Unity.
  2. Olhe a aba Game.
  3. Veja se o jogo aparece.
  4. Teste as teclas: A/D, setas esquerda/direita, Espaço, W.

Se o personagem não se mexer, pode ser que: o script de movimento ainda não exista, o personagem não esteja configurado, o foco não esteja na aba Game, ou o projeto use outro tipo de input.

Dica rápida

Clique dentro da aba Game antes de testar as teclas.

Fase 1 — Personagem

09🎨🕹️ Personagem

Como separar sprites de uma imagem

Se a IA gerar uma imagem com várias expressões juntas (idle | happy | sad | surprised | talking), você separa isso dentro do Unity.

  1. Importe a imagem pra Assets/Art/Characters.
  2. Clique na imagem no painel Project.
  3. No Inspector: Texture Type: Sprite (2D and UI), Sprite Mode: Multiple.
  4. Clique em Sprite Editor.
  5. Clique em Slice.
  6. Se as imagens estão em grade certinha, escolha Grid by Cell Size (ex: Cell Size X: 512, Y: 512).
  7. Clique em Slice, depois Apply.

Depois disso, ao expandir a imagem no Project, cada sprite aparece separado.

10🕹️ Personagem

Como colocar o personagem na cena

Se ainda não existir personagem:

  1. No Project, encontre o sprite do personagem.
  2. Arraste pra Scene.
  3. Renomeie pra Player.
  4. Adicione Rigidbody2D + BoxCollider2D ou CapsuleCollider2D (recomendado — mais arredondado, prende menos nas bordas).
  5. No Rigidbody2D: Gravity Scale: 3, Freeze Rotation Z: marcado (evita que ele tombe).
6

Fase 2 — Cenário

✦🏞️🕹️ Fase 2

O cenário tem mais que só o chão

Um cenário completo tem três tipos de peça, e cada uma se monta de um jeito diferente:

PeçaPrecisa de collider?Exemplo
Chão / plataformasSim — é nelas que o personagem pisaGround, plataformas flutuantes
Decoração de fundoNão — é só visual, ninguém colide com elaParedes, tubos, computadores, terminais, caixas
ObstáculosSim, mas como trigger (detecta sem travar)Energy barrier, laser — ficam na Fase Mecânica

Esta fase foca no chão/plataformas (seções abaixo), que é a parte que precisa de física e por isso é a mais delicada de configurar certo.

A decoração é bem mais simples — pra cada elemento (parede, tubo, computador, caixa):

  1. Arraste o sprite do elemento pra Scene.
  2. Posicione onde quiser com o Transform.
  3. Prontinho — sem componente extra, a menos que você queira que o personagem bata nela (nesse caso, trate como chão: adicione um BoxCollider2D).
Sobre profundidade (Sorting Layer)

Quando várias imagens se sobrepõem (ex: o personagem passando na frente de uma parede), o Unity decide o que aparece na frente usando Sorting Layer e Order in Layer, no componente Sprite Renderer. Número maior = aparece mais na frente.

11🕹️ Chão

Como saber onde está o chão

Na Hierarchy, procure nomes como Ground, Floor, Platform, Tilemap, Level, Terrain.

  • Chão simples → no Inspector: Transform, Sprite Renderer, BoxCollider2D.
  • Tilemap → Tilemap, Tilemap Renderer, Tilemap Collider 2D.
Em palavras simples

Box Collider 2D é um component (lembra da seção 6?) que você anexa num GameObject pra dar a ele uma área invisível de colisão. Não precisa ser igualzinho ao desenho — pode ser uma aproximação.

12🕹️ Chão

Como colocar chão na cena

Existem dois jeitos de fazer isso — escolha o mais simples pro momento.

Jeito 1 — Objeto simples com Sprite (mais fácil)

  1. No painel Hierarchy, clique com botão direito.
  2. Vá em 2D Object > Sprite.
  3. Renomeie pra Ground.
  4. No Sprite Renderer, coloque o sprite de chão.
  5. Posicione com o Transform (ex: Position X: 0, Y: -3, Z: 0).
  6. Adicione um collider: Add Component > Box Collider 2D.

Agora o chão tem “corpo” pro personagem pisar.

Jeito 2 — Usando Tilemap (melhor pra fases com blocos)

Tilemap é uma grade — pensa num quadriculado de papel — onde você "pinta" blocos repetidos (tiles) pra montar o cenário, tipo Lego. É melhor que o Jeito 1 quando você tem muitos blocos parecidos: paredes longas, chão irregular, várias plataformas.

O caminho, com cada passo explicado:

  1. Importar o tileset — arraste pra Assets a imagem com todos os blocos (ex: os 20 tiles do prompt da seção 31).
  2. Separar no Sprite Editor — igual à seção "Separar sprites": cada bloquinho da imagem precisa virar um sprite individual, fatiado em grade.
  3. Criar uma Tile Palette — é a "caixa de tintas": Window > 2D > Tile Palette > Create New Palette, depois arraste os sprites separados pra dentro dela.
  4. Pintar a fase — com a Tile Palette aberta, escolha um tile e clique na Scene pra "pintar" ele na grade, como um pincel.
  5. Adicionar o Tilemap Collider 2D — no GameObject Grid/Tilemap que foi criado automaticamente, adicione esse component pro personagem conseguir pisar nos blocos pintados.
Pra começar

No evento, se estiver começando, use o jeito simples primeiro: objeto com sprite + BoxCollider2D.

13🕹️ Chão

Como fazer o chão crescer

Depende de como o chão foi montado na sua fase:

Se o chão for um GameObject simples

Clique em Ground → Inspector → Transform. Pra deixar mais largo, mude o Scale X (ex: X: 10, Y: 1, Z: 1). Pra mais alto: Scale Y: 2. O collider geralmente cresce junto se estiver no mesmo objeto.

Se o chão for um Tilemap

Você não “estica” o chão — pinta mais bloquinhos usando a janela Tile Palette.

14🕹️ Chão

Como fazer o chão ficar fixo

Pra ficar parado, o chão não deve ter Rigidbody2D dinâmico. Ele precisa de Sprite Renderer + BoxCollider2D, e normalmente não precisa de Rigidbody2D.

Se tiver Rigidbody2D

Confira se está como Body Type: Static. O chão é o objeto parado; quem se mexe é o personagem.

Fase 3 — Hardware

🔌🔌 Fase 3

Hardware: visão geral

Enquanto as fases de Personagem e Cenário cuidam do que aparece na tela, esta fase cuida do que existe fisicamente na mesa: o ESP32 (uma placinha) e o OLED (uma telinha), que vão reagir ao jogo com carinhas.

ESP32 → placa física, conecta ao computador por USB OLED → telinha ligada no ESP32, mostra as expressões

Nesta fase, a pessoa de Hardware cuida de: ligar o OLED no ESP32, escrever o código que desenha as carinhas, e testar se ele funciona sozinho — antes de conectar com o Unity.

Os detalhes completos estão mais adiante

Ligações, código do ESP32/OLED e como testar tudo (sozinho e junto com o Unity) têm uma seção própria, mais pra frente neste guia, no grupo "ESP32 e OLED na prática". Aqui é só pra ter o panorama antes de seguir pras próximas fases.

8

Fase 4 — Mecânica

15🕹️ Personagem

Como colocar o personagem em cima do chão

Se o personagem aparecer flutuando ou enterrado no chão, é só ajustar a posição de um dos dois.

  1. Clique no personagem na Hierarchy.
  2. No Inspector, olhe o Transform.
  3. Ajuste a posição (ex: X: 0, Y: 0, Z: 0).
  4. Clique no chão e veja onde ele está (ex: Ground Y: -3).
  5. O personagem precisa estar acima (ex: Player Y: -1).

Se ele estiver dentro do chão, pode travar ou ser empurrado. Se estiver muito alto, vai cair até encostar.

Pra funcionar

Personagem precisa de Rigidbody2D + Collider2D; chão precisa de BoxCollider2D.

Como fazer o personagem andar

O projeto já traz o PlayerController

Tem um PlayerController.cs pronto em Assets/Scripts. Ele foi feito pra andar com os botões físicos do ESP32 — e já vem com um fallback de teclado/gamepad embutido, pra vocês testarem mesmo sem o controle. Você só precisa configurar.

Como funciona por dentro

No FixedUpdate, ele pergunta pro ESP32SerialReader se os botões ButtonRight/ButtonLeft estão pressionados pra andar, e usa o JumpPressed pra pular. Se não houver ESP32 (ou se o modo teclado estiver ligado), ele usa o teclado pelo New Input System. Detecta o chão com Physics2D.OverlapCircle numa Layer, e vira o sprite mudando o localScale.

Como configurar no Player

  1. Confirme que o Player tem Rigidbody2D e Animator — o script exige os dois ([RequireComponent]).
  2. Pro teclado funcionar como fallback, adicione um componente PlayerInput: em Actions, aponte pro asset InputSystem_Actions; em Behavior, escolha Send Messages (é assim que os métodos OnMove/OnJump são chamados).
  3. Configure a Layer Ground e o Ground Check (explicado na próxima seção) — o script usa isso pra saber quando pode pular.
  4. No Animator do Player, crie um parâmetro Bool chamado exatamente IsMoving — é o único que o script controla (liga a animação de andar).
Testando sem a placa

No Inspector do PlayerController existe uma opção Force Keyboard Fallback. Marque ela pra ignorar o ESP32 e mover só pelo teclado/gamepad — perfeito enquanto o hardware ainda não está pronto. Desmarque quando for usar o controle físico.

Se o personagem não andar

Confira, nessa ordem: pra testar no teclado, o Force Keyboard Fallback está marcado e o PlayerInput está configurado com Send Messages? A Layer Ground está no campo certo e o Ground Check está posicionado na base do personagem? O Animator tem o parâmetro Bool IsMoving com esse nome exato?

17🕹️ Personagem

Como fazer o personagem reconhecer o chão

O PlayerController usa Layer, não Tag

Pra saber quando pode pular, o script checa o chão com Physics2D.OverlapCircle numa Layer chamada Ground — e num pontinho (Ground Check) na base do personagem.

Passo 1 — criar/aplicar a Layer Ground

  2. No topo do Inspector, veja o campo Layer (é diferente do campo Tag, ao lado).

Passo 2 — criar o Ground Check

  1. No Player, crie um GameObject filho vazio (botão direito no Player → Create Empty).
  2. Renomeie pra GroundCheck e posicione ele bem na base do personagem (nos pés).
  3. Arraste esse GroundCheck pro campo Ground Check do PlayerController.

Um trigger não bloqueia o personagem fisicamente — ele só detecta quando outro collider passa por ele.

Quando o Player encosta, o Collectible avisa o GameManager (OnItemCollected), manda a carinha feliz pro OLED (SendTreasure), toca um efeito/som opcional e some. Referência:

referência · Collectible.cs (resumo)
private void OnTriggerEnter2D(Collider2D other)
{
    if (!other.CompareTag("Player")) return;

    GameManager.Instance?.OnItemCollected();      // conta o item
    ESP32SerialReader.Instance?.SendTreasure();  // carinha feliz no OLED

    // efeito e som são opcionais (campos no Inspector)
    Destroy(gameObject);
}

ESP32 e OLED na prática

✦🔌 Contexto

O que é o OLED e por que as carinhas importam

O componente

O OLED é uma telinha pequena (normalmente o modelo SSD1306, monocromática, com uns 128×64 pixels) ligada ao ESP32. É nela que aparecem as carinhas do fantasminha reagindo ao que acontece no jogo. Ela se conecta ao ESP32 por 4 fios (I2C): energia, terra, SDA e SCL — o passo a passo das ligações está na seção 24.

As carinhas são o coração do projeto: sem elas, o jogo seria só mais um jogo na tela. É a expressão no OLED que faz o jogo “sair da tela” e reagir no mundo real — quando você pega um item, uma telinha física ali do lado fica feliz.

No projeto, são três carinhas. O Unity manda uma letra pela Serial, o ESP32 recebe e desenha a expressão certa:

LetraCarinhaQuando aparece
Ineutra (O_O)jogo rodando normal
Tfeliz (^u^)pegou um item / venceu
Dassustada (>_<)bateu num obstáculo
Quer mais expressões?

Essas três são o suficiente pro jogo funcionar. Expressões a mais (surpresa, comemoração…) são uma personalização — dá pra criar novas letras/métodos no ESP32SerialReader e novas carinhas no firmware.

Duas “carinhas” diferentes — não confunda

As expressões podem aparecer em dois lugares: (1) no personagem dentro do jogo, que usa o sprite gerado no ChatGPT (seção 31, trocado pelo Sprite Renderer); e (2) na telinha OLED, onde as carinhas são desenhadas pelo código do ESP32. Podem ter o mesmo estilo pra ficar bonito, mas são feitas de formas diferentes.

✦🔌⚙️ Comunicação

O controle físico e as reações do OLED

O cabo USB é uma via de mão dupla: o ESP32 é um controle físico (você aperta um botão na placa e o personagem anda), e o Unity manda de volta a reação que aparece no OLED. No projeto, quem cuida disso é o script ESP32SerialReader.

CONTROLE → JOGO ESP32 manda os botões → Unity move o personagem JOGO → OLED Unity manda a reação → ESP32 desenha a carinha

ESP32 → Unity (os botões)

O ESP32 avisa quando um botão é pressionado e quando é solto:

MensagemSignifica
L1 / L0esquerda pressionada / solta
R1 / R0direita pressionada / solta
D1 / D0baixo pressionado / solto (reservado)
Upulo

Unity → OLED (as reações)

São só três carinhas, cada uma é uma letra:

LetraCarinhaMétodo no código
Ineutra (O_O)SendIdle()
Tfeliz / pegou tesouro (^u^)SendTreasure()
Dassustada / levou dano (>_<)SendDamage()

Do lado do Unity, mandar uma reação é só chamar o método pelo singleton do ESP32SerialReader:

enviar reação · C#
// carinha feliz (pegou item / venceu)
ESP32SerialReader.Instance?.SendTreasure();

// carinha de dano
ESP32SerialReader.Instance?.SendDamage();

// carinha neutra
ESP32SerialReader.Instance?.SendIdle();
Quer mais carinhas? (desafio extra)

O projeto vem com essas três (I, T, D). Se a equipe quiser expressões a mais — tipo comemorando ou surpresa —, dá pra criar novos métodos no ESP32SerialReader (ex: um SendCelebrating() que manda uma letra nova) e tratar essa letra no firmware do ESP32. É opcional, pra quem quiser ir além do mínimo.

25⚙️🔌 Serial

Comunicação Serial no Unity (ESP32SerialReader)

Configuração obrigatória do .NET no Unity

Por padrão o Unity não deixa usar SerialPort. Vá em Edit > Project Settings > Player > Other Settings, ache Api Compatibility Level e troque pra .NET Framework. Sem isso, System.IO.Ports.SerialPort nem existe e dá erro de compilação.

O projeto já traz o script ESP32SerialReader.cs em Assets/Scripts. Ele cuida das duas pontas da conversa: recebe os botões do controle e envia as reações pro OLED. Você não precisa criar — só configurar.

Como configurar

  1. Crie um GameObject vazio na cena e chame de ESP32SerialReader (ou coloque o script num objeto que já exista).
  2. Arraste o script ESP32SerialReader pra esse objeto.
  3. No Inspector, ajuste o Port Name: no Windows é algo como COM3; no Mac, algo como /dev/tty.usbserial-0001.
  4. Deixe o Baud Rate em 115200 (tem que bater com o ESP32).
O que ele oferece pros outros scripts

Pra ler os botões: as propriedades ButtonLeft, ButtonRight, ButtonDown (true enquanto pressionado) e JumpPressed (true por um frame no pulo). Pra mandar reação: os métodos SendIdle(), SendTreasure() e SendDamage(). Tudo pelo singleton ESP32SerialReader.Instance.

Referência das partes principais do script (o arquivo completo está no projeto):

referência · ESP32SerialReader.cs (resumo)
public class ESP32SerialReader : MonoBehaviour
{
    public static ESP32SerialReader Instance { get; private set; }

    // Estado dos botões, lido pelo PlayerController
    public bool ButtonLeft  { get; private set; }
    public bool ButtonRight { get; private set; }
    public bool ButtonDown  { get; private set; }
    public bool JumpPressed { get; private set; }

    // Reações enviadas pro OLED
    public void SendIdle()     => WriteChar('I');  // (O_O)
    public void SendTreasure() => WriteChar('T');  // (^u^) feliz
    public void SendDamage()   => WriteChar('D');  // (>_<)

    // ...abre a porta, lê L1/L0/R1/R0/D1/D0/U numa thread, e escreve as letras...
}
Sobre a porta e o baud

O baud rate precisa ser igual no Unity e no ESP32 (115200). Se a porta estiver errada, o jogo abre do mesmo jeito — só não conversa com a placa.

✅ Teste agora

Aperte Play e olhe a Console: deve aparecer [ESP32] Conectado em .... Se aparecer um aviso de falha ao abrir a porta, é normal caso o ESP32 ainda não esteja ligado ou a porta esteja diferente — ajuste o Port Name no Inspector.

26🔌 ESP32

Código do ESP32/OLED

Este é o lado do ESP32 — o código que a equipe de Hardware escreve e sobe pra placa (não vem no projeto Unity). Pra OLED SSD1306 costuma-se usar as bibliotecas Adafruit SSD1306 e Adafruit GFX (instaláveis pelo Library Manager). A SSD1306 cuida da comunicação com o display; a GFX fornece texto, linhas e formas.

Arduino IDE ou VS Code — os dois servem

A ferramenta é só o jeito de passar o código pra placa (o ESP32). Pode usar a Arduino IDE (mais simples pra começar) ou o VS Code (com a extensão PlatformIO). Os dois fazem a mesma coisa; este guia mostra os passos pela Arduino IDE, mas no VS Code o caminho é equivalente. Use o que a equipe preferir.

Ligações comuns (OLED I2C no ESP32):

OLED VCC → 3V3 OLED GND → GND OLED SDA → GPIO21 OLED SCL → GPIO22
Observação

No ESP32 genérico, GPIO21 (SDA) e GPIO22 (SCL) são o padrão nos exemplos I2C, mas isso pode variar conforme a placa.

O Unity manda uma LETRA, não uma palavra

O ESP32SerialReader envia um único caractere: 'I' (neutra), 'T' (feliz) ou 'D' (dano). Então o firmware do ESP32 precisa ler um caractere e reagir a essas três letras — e mandar de volta os botões (L1/L0, R1/R0, U…).

Ler a letra que veio do Unity e escolher a carinha:

esp32.ino
if (Serial.available()) {
  char c = Serial.read();   // lê um caractere

  if (c == 'I') {
    showIdle();       // (O_O)
  } else if (c == 'T') {
    showTreasure();   // (^u^) feliz
  } else if (c == 'D') {
    showDamage();     // (>_<)
  }
}
✦🔌 Hardware

Como desenhar as carinhas no OLED

Essa parte é da pessoa de Hardware. A telinha do OLED é monocromática (só acende ou apaga cada pixel, geralmente branco no fundo preto), então as carinhas são desenhadas com formas simples — círculos pros olhos, linhas pra boca — e não fotos coloridas.

Passo 1 — Preparar a tela

No topo do arquivo .ino, você inclui as bibliotecas e cria o objeto do display. A Adafruit_GFX dá as funções de desenho (círculo, linha, retângulo) e a Adafruit_SSD1306 conversa com a tela.

topo do esp32.ino
#include <Wire.h>
#include <Adafruit_GFX.h>
#include <Adafruit_SSD1306.h>

#define LARGURA 128
#define ALTURA 64

Adafruit_SSD1306 display(LARGURA, ALTURA, &Wire, -1);

void setup() {
  Serial.begin(115200);
  display.begin(SSD1306_SWITCHCAPVCC, 0x3C); // 0x3C é o endereço comum do OLED
  showIdle(); // começa com a carinha parada
}

Passo 2 — A regra das 3 etapas

Sempre nessa ordem

1. display.clearDisplay() — apaga o que estava antes. 2. desenha a carinha nova (olhos, boca…). 3. display.display() — só aí a tela realmente mostra. Se esquecer o passo 3, nada aparece!

Passo 3 — Uma função por carinha

Cada expressão é uma função. Repare que muda só a boca — os olhos são quase sempre iguais. Coordenadas: (x, y) começa no canto de cima à esquerda.

as carinhas
// olhos que quase todas as carinhas usam
void desenhaOlhos() {
  display.fillCircle(45, 25, 6, SSD1306_WHITE); // olho esquerdo
  display.fillCircle(83, 25, 6, SSD1306_WHITE); // olho direito
}

// neutra: boca reta
void showIdle() {
  display.clearDisplay();
  desenhaOlhos();
  display.drawLine(50, 45, 78, 45, SSD1306_WHITE);
  display.display();
}

// feliz / tesouro: boca sorrindo (linhas subindo nas pontas)
void showTreasure() {
  display.clearDisplay();
  desenhaOlhos();
  display.drawLine(50, 42, 64, 50, SSD1306_WHITE);
  display.drawLine(64, 50, 78, 42, SSD1306_WHITE);
  display.display();
}

// dano: boca virada pra baixo
void showDamage() {
  display.clearDisplay();
  desenhaOlhos();
  display.drawLine(50, 50, 64, 42, SSD1306_WHITE);
  display.drawLine(64, 42, 78, 50, SSD1306_WHITE);
  display.display();
}

Essas três (showIdle, showTreasure, showDamage) cobrem o protocolo do projeto. Se quiser expressões extras (surpresa, comemoração…), é só criar mais funções no mesmo padrão — ex: uma boca em "O" com display.drawCircle(64, 46, 6, SSD1306_WHITE) — e mandar uma letra nova do Unity pra elas.

Passo 4 — Ligar as letras às funções

No loop(), você lê a letra que veio do Unity e chama a carinha certa — é a versão completa do trechinho da seção 24:

loop() do esp32.ino
void loop() {
  if (Serial.available()) {
    char c = Serial.read();

    if (c == 'I')      showIdle();
    else if (c == 'T') showTreasure();
    else if (c == 'D') showDamage();
  }

  // aqui também vai a leitura dos botões pra mandar L1/L0/R1/R0/U pro Unity
}
Quer usar imagens no lugar de formas?

Dá pra desenhar uma carinha feita de imagem com display.drawBitmap(...). Aí você converte um PNG preto e branco pequeno num vetor de bytes usando um site como o image2cpp, cola o vetor no código e chama no lugar dos olhos/boca. É mais bonito, mas dá mais trabalho — pro evento, começar com as formas simples é o caminho mais tranquilo.

Se a tela ficar preta

Confira: esqueceu o display.display()? O endereço 0x3C está certo (algumas telas usam 0x3D)? Os fios SDA/SCL estão nos GPIO certos (seção 24)?

✦ Ferramenta do evento

O que é o Kiro e como ele entra no projeto

Como este é um evento da AWS, a ferramenta de código será o Kiro — um editor com inteligência artificial que a AWS criou. Ele é o assistente de código de vocês: em vez de escrever tudo em C# na mão, você conversa com ele em inglês descrevendo o que quer, e ele lê o projeto, entende o que já existe e escreve ou edita os arquivos por você — explicando o que mudou.

Em palavras simples

Pense no Kiro como uma pessoa da equipe que sabe programar muito bem e senta ao seu lado. Você pede (“crie um item que soma pontos”), ela faz e te explica. Você continua no comando — quem decide o que pedir, testa e aprova é você.

Como ele funciona no fluxo

O Kiro mexe direto nos arquivos da pasta do projeto Unity. O ciclo é sempre esse:

Você pede em inglês → Kiro lê o projeto e edita os scripts (.cs) → Você volta pro Unity → Unity recompila sozinho → Você aperta Play e testa → deu certo? pede a próxima

Quem usa o Kiro (e quem não)

O Kiro é a ferramenta principal pra código: as pessoas de Gameplay (scripts, regras, pontuação) e também dá pra usar no firmware do ESP32. Pra arte (personagem, expressões, cenário), a equipe pode escolher entre ChatGPT ou Kiro — os prompts da seção 31 são só texto, então funcionam nos dois; use o que a equipe preferir ou já tiver configurado.

Uma dica prática

Os prompts da seção 31 são só texto, então colam do mesmo jeito no ChatGPT ou no Kiro. Se for gerar pelo Kiro, só confirme que a geração de imagem está disponível na configuração que vocês vão usar no evento.

A regra que evita dor de cabeça

Peça uma coisa de cada vez, teste, e só então peça a próxima. Se você despejar cinco pedidos num prompt só e algo quebrar, fica quase impossível achar onde. Prompt curto e específico > prompt gigante.

Sempre comece assim

Antes de pedir qualquer mudança, deixe o Kiro conhecer o projeto sem mexer em nada (é o primeiro prompt da próxima seção). Assim ele responde com base no que realmente existe, não em suposições.

35 Roteiro

Roteiro do dia do evento, literalmente

A ordem abaixo segue exatamente a ordem deste guia — se travar em algum passo, é só voltar pra seção correspondente.

  1. Decidir os papéis da equipe (quem faz o quê).
  2. Abrir GitHub e clonar o repositório (Git Bash ou Kiro).
  3. Confirmar que a pasta tem Assets, Packages e ProjectSettings — ela já é o projeto Unity.
  4. Abrir essa pasta no Unity Hub.
  5. Abrir a cena principal e clicar Play, só pra testar.
  6. Separar os sprites do personagem e colocar na cena (Fase Personagem).
  7. Encontrar e configurar o chão (Fase Cenário).
  8. Ver a visão geral da Fase Hardware.
  9. Configurar o PlayerController (ou criar um do zero) e testar o movimento.
  10. Criar os itens coletáveis e testar a pontuação.
  11. Criar os obstáculos e testar a colisão.
  12. Criar a HUD e o ponto de vitória.
  13. Montar a fase mínima com os itens e obstáculos em ordem.
  14. Ligar o OLED no ESP32 e testar sozinho pelo Serial Monitor.
  15. Subir o firmware com as expressões.
  16. Fechar o Serial Monitor e testar o Unity enviando comandos pro ESP32.
  17. Testar todos os eventos do jogo reagindo no OLED.
  18. Personalizar personagem, cenário, obstáculos e itens.
  19. Testar tudo de novo, do zero.
  20. Atualizar a documentação de comandos.
  21. Fazer a entrega final. 🎉
